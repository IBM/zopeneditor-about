Agent Mode

In version 1.102 of VS Code, Microsoft introduced Agent Mode. Among other features, Agent Mode added support for the Model Context Protocol (MCP), which enables AI models used in VS Code access to external tool APIs and services. MCP is evolving into an open standard. It is implemented by many vendors and projects, and is supported by various AI clients such as IBM Bob or VS Code's GitHub Copilot Chat.

The MCP protocol is implemented by VS Code, IBM Bob™ and many of its extensions as servers that provide access to data required by AI models to deliver accurate answers or to reach resources that are otherwise unavailable. Examples of MCP server functions include accessing your workspace files for review and modification, accessing your Git history to learn about recent changes and their reasons, and accessing resources on the internet, such as the latest documentation for tools and services that the AI model was not originally trained on. MCP tools can also perform active functions, such as executing VS Code tasks or running command-line operations in the terminal.

Z Open Editor adds its own set of MCP capabilities in the form of tools and resources that you can use with an AI chat. By leveraging Z Open Editor's Zowe™ APIs, the AI interacts with your z/OS® system and accesses data sets, UNIX System Services files, and jobs. For example, you can ask the AI to list the members of a specific data set, fetch files that match a search pattern and compare them with files open in the current workspace, list your jobs and group them by status, or submit a new job using JCL selected in the editor or stored in a data set. In all cases, the AI displays the commands it intends to execute, shows the parameters, and requests your permission before proceeding. Another set of tools works with Z Open Editor's configuration files to help diagnose setup issues. You can ask the AI to check the editor's log file and ZAPP settings to determine why a copybook cannot be found for the currently open COBOL program. You can also inquire about your Zowe profiles and which defaults apply. Z Open Editor also provides MCP tools for its language capabilities such as control and data flow graphs, as well as key integrations including IBM® Dependency-Based Build, IBM ZCodeScan and IBM Wazi Deploy.

See the Z Open Editor MCP Reference page for a list of available MCP tools and resources. This list expands with each release as new tools are added incrementally.

IBM Bob™ Premium Package for Z

IBM Z Open Editor is part of the IBM Bob™ Premium Package for Z. This premium package extends IBM Bob by delivering agents designed to help you analyze, explain, and develop your mainframe applications with greater efficiency and confidence. Its key use cases at moment are:

  • Accelerate mainframe development with AI and automation.
  • Modernize applications and build new enterprise services.
  • Extract business logic to better understand legacy code.

Z Open Editor is included in this package, unlocking almost all advanced capabilities when running as part of it. It also provides access to its MCP server and other dedicated integrations that contribute directly to Premium Package modes and skills. See the IBM Bob documentation for more details.

When Z Open Editor runs as part of IBM Bob Premium Package for Z, it automatically performs an entitlement check. The results are reported on the Welcome page together with the Advanced Capabilities unlocked. A series of tools are then registered with IBM Bob and automatically used in Premium Package modes such as ZCodeScan in Z Code mode or control and data flow tools in Z Architecture mode. For these tools, MCP server registration described later in this section is not needed.

To access the full set of MCP tools provided by Z Open Editor, perform that registration as well. Z Open Editor's MCP tools are available to IBM Bob Premium Package for Z, IBM Developer for z/OS Enterprise Edition, and IBM Application Delivery Foundation for z/OS users.

How to use Agent Mode

As with any business use of AI, it is important to understand the implications. When you run MCP server tools and access resources, you grant the AI access to these data sources. This means content loaded from your z/OS system becomes available to the AI and is collected by the owners of the AI's language model. The AI can also create or delete files and data sets, execute jobs, and perform other operations.

It is therefore important to use these features only with AI setups that you trust and that are approved for use in your organization.

It is recommended not to enable "auto approve" features, which enable MCP tools to perform operations without user confirmation, to avoid accidental loss of data. All tools and resources provided by Z Open Editor must be explicitly enabled before they can be used. Enable them only when the preceding preconditions are met.

As a VS Code user, review the Security section in the VS Code documentation about Agent Mode, which describes additional potential security risks. If you use the IBM Bob IDE, review this alternative page for security information.

The main switch for Z Open Editor's MCP feature is the user setting: zopeneditor.mcp.enabled. It is disabled by default.

The following sections describe setup steps for two alternative AI chat client solutions tested with Z Open Editor's MCP server. They explain how to use Z Open Editor MCP in:

  • IBM Bob IDE
  • VS Code with GitHub Copilot Chat

The Z Open Editor MCP tools also work with any other client that implements MCP and is used within an organization. Refer to the documentation for those client's for details on how to configure third-party MCP servers by using the "http-streaming" protocol.

Setup for IBM Bob IDE

IBM Bob is IBM's AI-powered development partner. It comprises standalone editor clients that provide a VS Code look-and-feel as well as a terminal-centric shell. Both variants can be configured to work with Z Open Editor's MCP tools. For detailed documentation of Bob's MCP support, see here for the IDE variant and here for the shell variant.

For both cases, a configuration file is needed to provide the details about Z Open Editor's MCP server to Bob. The format of these files is very similar to the VS Code MCP configuration file and almost identical to the Roo Code file described later in this section. Note that because Z Open Editor's MCP server runs as part of the Z Open Editor Bob/VS Code extension, you still must run a Bob IDE with the MCP configured if you want to utilize its MCP features in Bob Shell.

  1. If you are an IBM Bob Premium Package for Z user, Z Open Editor is automatically downloaded and installed for you. This does not apply to regular IBM Bob and IBM Developer for z/OS Enterprise Edition users. They must install Z Open Editor into IBM Bob either through its integration with the Open-VSX Marketplace or manually using the extension file downloaded from the Mainframe downloads site, as documented in Getting Started. The installation process is identical to that for VS Code.

  2. If you are an IBM Bob Premium Package for Z user and you are prompted to log in to Bob at startup then you need to reload the editor after finishing the log in procedure to go through the entitlement checks again. This is also necessary if you are part of more than one Bob Premium team and switch between teams in Bob Settings. The easiest way to do such a reload is to open the command palette with cmd-shit-p / ctrl-shift-p and select the command "Developer: Reload Window".

  3. Load the source code workspace that you want to use with the MCP server. Each workspace requires its own .bob/mcp.json file with a unique server name and port, even when using multi-root workspaces. Do not create a global mcp.json file because the MCP tools lack the context of the specific source code workspace that you are working on.

  4. To create an empty file, open the Bob Settings editor from the IBM Bob Chat view and select the "MCP" page. Click the + icon in the top-right corner. When prompted for the file location, select your workspace name for the "Configuration Scope" value. The empty file opens in the json editor.

  5. If you already have a .bob/mcp.json file in your workspace folder, or you just created one in the previous step, add the Z Open Editor entry as shown later in this section. The server name must contain the string "zopeneditor" or the entry is ignored. You can concatenate it with the workspace name, for example. If you are using multiple workspaces in a multi-root configuration or in multiple open editor windows, ensure that each workspace uses a unique name following "zopeneditor-" and unique port number listed in its file.

    {
  "mcpServers": {
    "zopeneditor-sample": {
      "url": "http://localhost:3005/mcp",
      "type": "streamable-http",
      "timeout": 600
    }
  }
}

  6. Enable Z Open Editor's MCP server with the user setting: zopeneditor.mcp.enabled.

  7. Reload the VS Code window to ensure all components are initialized and restarted.

  8. From the IBM Bob Chat view, open the Bob Settings editor again and switch to the "MCP" page.

  9. Ensure that the Z Open Editor MCP server appears with the same name specified in the configuration file.

  10. If the Z Open Editor's MCP server entries show as disconnected, click the Refresh icon in the top-right corner. Repeat this action until the server shows as connected.

  11. After the refresh, the complete list of MCP tools provided by Z Open Editor appears when you click the server entry in the list.

  12. Disable the tools you do not want to use with slider buttons. Disabling tools helps minimize token consumption because MCP tool metadata is included in the AI chat’s context window.

  13. Manage the "Always allow" state for each tool using the check boxes. Do not automatically enable tools that perform write operations locally or on z/OS.

  14. The results of these selection are appended to the .bob/mcp.json file and can be committed through Git so your workspace is shared with team members.

  15. Run Bob in the "Agent Mode" or, when using the Premium Package, in "Z Code" or "Z Architect" mode. These modes are preferred for using Z Open Editor MCP tools with Bob.

Setup for VS Code with GitHub Copilot Chat

GitHub Copilot Chat is Microsoft’s default AI‑chat solution for VS Code and enables Agent Mode. It requires a GitHub account and a subscription after the free request limit or trial period ends. The extension can be configured to work with different large language models from a list of providers. Detailed setup instructions are available in the VS Code documentation.

Steps to enable and test Z Open Editor's MCP server:

  1. Install and configure the GitHub Copilot Chat extension in VS Code and sign in with your GitHub account.

  2. Enable Agent Mode in VS Code with the user setting chat.agent.enabled.

  3. In the Chat panel's lower-left corner, change the mode drop-down from its default "Ask" value to "Agent".

  4. In the model selection drop-down next to it, select the large language model you are authorized to use. Use "Manage Models.." for additional configuration options if needed.

  5. Load your source code workspace and create an MCP configuration file by adding an entry in the workspace's .vscode/mcp.json file. If you use a multi-root workspace or multiple editor windows, create one in each workspace with unique name postfixes and port numbers to set the correct context for the AI. Alternatively, run the "MCP: Add Server" command to use prompts to create and edit the file.

    {
  "servers": {
    "zopeneditor-sample": { "url": "http://localhost:3005/mcp" }
  }
}

  6. Ensure the server name starts with zopeneditor followed by a workspace-specific prefix, as shown in the preceding example. This enables you and the AI to distinguish between different servers running in multiple workspaces. Each server you run in parallel requires a unique port number and name. Errors appear in the "Z Open Editor MCP" Output panel if a server cannot start because its port is already in use.

  7. Enable Z Open Editor's MCP server with the user setting: zopeneditor.mcp.enabled.

  8. Reload the VS Code window to initialize and restart all components.

  9. Check the Z Open Editor output panel for an INFO message reporting that the MCP feature is enabled (search for message id "CRRZG5452I").

  10. Switch to the Z Open Editor MCP output panel and observe the startup messages. Set Z Open Editor's logger to the DEBUG level to view detailed information. The log reports that it found the mcp.json file and started the server using the workspace configuration.

  11. Switch back to the Chat panel in VS Code and click the tools icon at the bottom-right.

  12. In the panel that appears at the top, the list of MCP tools is displayed. Under the "Built-In" section, locate the entry that matches the server name specified in the mcp.json file. Expand it to view all MCP tools provided by Z Open Editor.

  13. Alternatively, type /list in the Chat panel to obtain the list of tools in the Chat's output.

  14. If the tools do not appear, check the Output panels again for errors reported.

  15. Execute any tool directly in the chat by using # with code completion on the tools list. For example, typing #zowe-list-all-profiles lists your profiles, assuming Zowe profiles are defined.

Configure and test Wazi Deploy tools

The Wazi Deploy Evidence MCP tools enables developers to query and analyze deployment evidence from z/OS systems through natural language prompts in Z Open Editor. These tools require some additional setup steps to the general steps listed above.

Prerequisites

Before testing, ensure the following:

  • Wazi Deploy is installed on the target z/OS system.
  • A Zowe profile for this system is selected in the status bar as the default.
  • The wazideploy-evidence executable is available in the PATH (preferably at the beginning of the PATH), or the absolute path to its location is known.
  • A configured evidence dataFolderexists on your z/OS system and contains valid evidence files.

In the Z Open Editor user settings, add values that match your Wazi Deploy configuration:

"zopeneditor.wazideploy.evidence": {
  "dataFolder": "/u/ibmuser/projects/deploy/data",
  "indexFolder": "/tmp/ibmuser_evidences_index",
  "executablePath": "/global/opt/pyenv/gdp/bin/wazideploy-evidence",
}

These settings are also available in the Settings GUI under "Zopeneditor > Wazideploy: Evidence".

  • dataFolder (required) - Path to the directory containing valid evidence files.
  • indexFolder (required) - Path to a writable directory for the evidence index.
  • executablePath (optional) - Full path to the wazideploy-evidence executable. Use this only if the executable is not available in the system PATH or is located in a non-standard directory.

When to use executablePath

  • If wazideploy-evidence is not set or available in the PATH
  • If multiple versions exist and a specific one must be used
  • If installed in a custom or non-standard directory

If wazideploy-evidence is already available in the PATH, this property can be omitted.

First-Time Setup

For initial setup, execute the following natural language prompts in your chat client:

Prompt: "Check Wazideploy evidence version"

or run the #wazideploy-check-version MCP tool directly

The version check returns a version number (for example, 3.0.6) with a build date. Review the "Z Open Editor MCP" output view for logging information and errors if the expected result is not returned.

Prompt: "Index the Wazideploy evidence folder"

or run the #wazideploy-index-evidence-folder MCP tool directly

The tool indexes the dataFolder successfully. Review the Output view for details of the operation and for troubleshooting information.

Note: To get the "type," evidences must be from version 3.0.7.1 or later; otherwise, you might see the "none" extension.

Configure Dependency Based Build (DBB) tools

The MCP server holds tools for interacting with a remote DBB CLI for the purposes of troubleshooting pipeline builds. To use them, you need to follow the configuration steps listed here.

Prerequisites

1. Install DBB on a remote system

Install DBB using the documentation found at Installing and configuring DBB.

2. Create a Db2® configuration file

If you do not already have a Db2 configuration file, create one on your remote environment and store the URL to your Db2 JDBC metadata store inside of it. This is a Java .properties file configured with a line similar to url=jdbc:db2://example.com:5040/DBB1

3. Create a Db2 password file

If you do not already have a Db2 password file, create one on your remote environment. Learn how to do so in the DBB documentation on Encrypting metadata store passwords.

4. Configure a Zowe RSE profile

Configure a Zowe RSE profile that connects to your remote environment and set it as your default profile.

First-Time Setup

To use the tools that access a DBB metadata store like dbb-get-build-result, you must configure the following VSCode settings:

"zopeneditor.dbb.metadataStore": {
 "db2User": "USERNAME",
 "db2Config": "~/db2-metadata-store.properties",
 "db2PasswordFile": "~/encryptedPasswordFile.pw"
}

This information is passed into the tool directly, so it is not seen by any AI agents that call it for privacy reasons.

If you include the buildGroup setting, the tool call uses it by default. If not, you must pass one to your agent when making the call.

"zopeneditor.dbb.metadataStore": {
 "buildGroup": "MortgageApplication-main"
}

If you include the localAttachmentsDir setting and call the dbb-download-build-result-attachments tool, the tool uses it by default. If not, you must pass one to your agent when making the call.

"zopeneditor.dbb.metadataStore": {
 "localAttachmentsDir": "/path/to/create/local/dir"
}

Examples of using the MCP tools

After the MCP server starts, you can access it from the Chat panel. Refer to the Z Open Editor page for a list of available tools, you can combine them with tools from other MCP servers and with the built‑in tools. Here are few things you can try. Note, the responses you receive depend on the language model you use and on other factors.

Try some of the following examples, if a valid Zowe profile is configured in your workspace that can be used to access z/OS data sets, files, and jobs through Zowe Explorer. Replace the example names with your own.

  • "List the members of the data set USER.SAMPLE.COBOL that start with the prefix SAM". The AI prompts you for permission to execute the MCP tool "zowe-get-members" with the data set name you provided. After you approve the execution, the tool returns the list of members that start with SAM. Expand and examine the tool entry in the chat to see the full response. You see that the AI reduced the full list of members it received to the members that match the prefix.

  • Open a modified version of one of the programs listed in the previous step and ask "Download the member with the name SAM1 and compare to the currently open COBOL program in my editor." The chat asks for approval to execute the "zowe-download-member" tool. After approval, the tool fetches the member and the AI provides a detailed explanation of the differences. The more the large language mode is trained on COBOL, the richer the explanation. If your are looking for a sample program, use the files "COBOL/SAM1.cbl" and "multiroot/sam/SAM1.cbl" from our samples repository. They are different versions of the same application.

  • "What are the jobs of user IBMUSER grouped by status?" The AI returns a list of job names organized in sections by status.

  • Use Zowe Explorer to open a JCL file from one of your data sets in Z Open Editor and ask "Submit the JCL in the currently open editor as a job." It will prompt for approval of the "zowe-submit-job-dataset" tool with the correct data set name. If you would have opened a local file it would have used the tool "zowe-submit-job-jcl" instead. After it ran the command it will show you the job id, name, and status of your job. It might even ask you to retrieve the job output or current status.

  • Do retrieve the outputs from your job by replying "Yes, retrieve the spool files for me." which will result into the list of files available, asking you which one you want to see. Reply with "Show me SYSOUT", which it then retrieves and displays in the chat. If the output contained error messages, then some AI models will even explain the reasons for why things might have gone wrong. If you ask "Fetch SYSOUT and open it in a VS Code editor tab", it would do that as well after asking you for permission, if it is allowed to save the content in a workspace file first.

  • "Fetch my .profile file from my UNIX home directory and open it in the editor. My userid is ibmuser.". The AI deduces the home directory "/u/ibmuser", asks for approval, downloads the file, and opens it in VS Code. If the deduction is incorrect, reject the request and refine the prompt with more specifics.

In addition to Zowe interactions, you can use tools and resources provided by Z Open Editor's MCP server to query your configuration.

  • "List my Zowe profiles and the profiles defined as default for Z Open Editor.". The AI executes two MCP tools, "zowe-list-all-profiles" and "zopeneditor-get-default-zowe-profile", then lists all profiles found in your personal Zowe team configuration files together with the default profile used by Z Open Editor.

  • "What are my ZAPP property groups for this workspace for COBOL and HLASM?". AI calls "zapp-get-property-groups" twice and returns a summary for each language, assuming the property groups are defined in your ZAPP file.

  • Open a program where some copybooks cannot be found and ask: "Using the zopeneditor.log resource and the ZAPP property groups, analyze why copybook include files could not be found for the currently open program?" Depending on the language model, the AI automatically uses the MCP resource defined by Z Open Editor MCP or prompts you to execute the tool "zopeneditor-read-the-log" together with the previously retrieved ZAPP information. It then provides a detailed breakdown of the search process, listing locations that succeeded, locations that failed, and user settings that might have influenced the search.

  • To use the Z Open Editor log file as a resource in GitHub Chat, click the "Add Context..." button in the chat panel, select the file under "MCP Resources" in the prompt at the top of the screen, and confirm. The log file then appears next to the open program as context for your questions. You can use it to find various types of information from a DEBUG-level log file, such as details about your Zowe configuration, the version of Java used by the editor, user setting values, and more.

  • Open a workspace that contains many COBOL files that must have sequence numbers added to them and ask: "Can you add sequence numbers to all the COBOL files matching the following pattern: COBOL/SOURCE/**". The files matching that pattern are then opened and have sequence numbers added to them. The changes are not saved by the tool to enable the user to confirm them. There are also tools to help if you'd like to remove sequence numbers or format a file instead, using your Z Code Format file.

  • You can also run ZCodeScan on many programs at once in your workspace using the same file patterns as shown above. "Run a scan of all the programs in the folder COBOL/SOURCE and describe the results in detail." Z Open Editor's MCP tool finds all the programs in the folder or matches a provided pattern, runs a code scan on each, and then aggregates the results into one report. You can ask follow-up questions such as "Can you explain the first issue in more detail and provide examples for how to fix it?" Many large language models then suggest how to perform the fix and guide you step-by-step.

There are also tools available for COBOL and PL/I program analysis.

  • Open a COBOL program and ask: "Using data flow information, what is the impact of increasing the length of CURRENT-YEAR from PIC 9(2) to PIC 9(4)?" The AI calls the data flow tool to trace where the data item is assigned, transformed, and referenced throughout the program. With that metadata, it estimates the scope of the change and recommends required modifications, reducing the risk of introducing truncation or format-related defects.

  • To analyze program structure, ask: "Using control flow information, which paragraphs call 730-READ-CUSTOMER-FILE and can this be optimized?" The AI calls the control flow tool to map how sections and paragraphs are connected across the program. This helps identify opportunities to simplify or streamline the program flow.

You can combine Z Open Editor’s capabilities with the VS Code built‑in MCP tools listed earlier.

  • If Z Open Editor reports syntax errors in your current program, you can use the integration with VS Code and its built-in MCP tools to resolve the issues reported in VS Code's Problems View. You can ask "Can you explain the issues listed in the Problems view for COBOL developers and suggest how to fix them?" If your large language model is trained on COBOL, it provides a detailed description of the problems found and suggests how to fix them.

  • "Who made the last few changes on this program looking at the Git history?". The AI generates a command line that queries the local Git repository, retrieves recent commits, and displays their descriptions.

  • "Can you find other programs in the workspace that have the same ACCEPT-DATE and ACCEPT-TIME statements that need fixing?". The AI uses the context of the previous interactions, runs a VS Code Search to find similar statements in other files in your workspace and lists the matching files with line numbers.

  • After you fixed the issues found by ZCodeScan you can ask to build the programs you fixed to ensure they still compile correctly with "Run a user build of the currently open program".

  • If a DBB pipeline build (not a user build) fails on a remote machine, you can ask the AI to "get the last failed build result". If the desired build result is not the most recent failure, you can ask it to "get the build result <result-label>". You may have to configure settings to access your metadata store if you have not already done so.

  • If the error is not clear in the build result and information from the build logs would help with troubleshooting, you can ask it to "download the attachments of the build result that was just fetched".
